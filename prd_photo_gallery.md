## 2. 需求背景

### 2.1 背景

用户希望制作一个用于展示个人照片的网页，当前已完成初步页面代码和关键页面设计，包括 Years、Locations、Photos、Admin 等页面。网站后续需要部署到云端，并支持外部访问；域名配置后续处理，当前优先级较低。

当前页面已具备基础照片分层展示能力，但仍存在以下问题：

1. 前台展示结构、排序规则、空年份隐藏逻辑尚需完善。
2. Photo 层级需要更适合照片浏览，不应显示照片文件名。
3. 后台管理页面目前直接暴露较多表单，需要优化为按钮触发式操作。
4. 需要增加后台账号密码登录、照片信息修改、照片删除、智能排序等能力。
5. 前台不应暴露 Admin 入口，避免普通访问者进入后台。

### 2.2 目标

搭建一个可本地运行、后续可云端部署的个人照片展示网站，实现：

1. 按 **Years → Locations → Photos** 的结构展示照片。
2. 后台支持上传照片、设置年月日时间、设置地点、修改照片信息、删除照片。
3. 前台展示页面具备液态玻璃视觉效果，并能根据数据量自适应布局。
4. Photo 层级支持基于 CLIP 视觉模型 + TSP 算法的智能排序。
5. 支持照片全屏查看、上一张 / 下一张切换，并在同一标签页内完成。
6. 前台隐藏后台入口，后台通过固定隐藏路径 `/admin` + 账号密码访问。

### 2.3 效果评估

| 指标 | 目标 |
|---|---|
| 前台访问 | 用户可通过网页正常浏览照片 |
| 展示结构 | 支持 Years、Locations、Photos 三层结构 |
| 后台管理 | 单管理员可登录并完成上传、修改、删除 |
| 排序效果 | 同一相册内照片可通过智能排序优化视觉过渡 |
| 页面体验 | Photo 页面紧密排列，不显示照片名称 |
| 安全性 | 前台不展示 Admin 入口，后台需登录访问 |

## 4. 功能需求详述

## 4.1 前台 Years 页面

### 操作者

普通访问用户。

### 前置条件

1. 网站可正常访问。
2. 后台已上传至少一张照片。
3. 照片已配置时间和地点。

### 需求说明

Years 页面是照片展示的一级入口，展示所有已上传照片对应的年份。

### 展示规则

1. 仅展示存在照片的年份。
2. 没有照片的年份不显示。
3. 年份卡片按年份排序，默认建议倒序展示，即最新年份在前。
4. 每个年份卡片展示：年份、该年份照片数量、可选封面图。
5. 封面图默认取该年份第一张照片或排序后的第一张照片。
6. 年份卡片背后的液态玻璃容器需要根据年份卡片数量自动调整大小。
7. 容器大小应适配年份卡片的排列结构：
   - 1～3 个年份：容器高度较小；
   - 4～6 个年份：容器高度增加；
   - 7 个以上年份：容器继续扩展或容器内部滚动。

### 交互说明

1. 点击某个年份卡片，进入该年份的 Locations 页面。
2. 底部导航中显示 Years、Places、Photos 等前台入口。
3. 前台底部导航不显示 Admin。

## 4.2 前台 Locations 页面

### 操作者

普通访问用户。

### 前置条件

1. 用户已点击某个年份。
2. 该年份下存在至少一个地点。

### 需求说明

Locations 页面用于展示某一年份下的地点相册。

### 展示规则

1. 地点为单一文本，例如“山东”“香港”“京都”。
2. 同一年份下的地点需要按时间排列。
3. 时间排序依据为该地点下照片的拍摄 / 设置时间。
4. 后台上传时设置完整年月日，但前台进入 Location 层级后，时间显示仅展示“月日”。
5. 地点卡片展示：地点名称、照片数量、可选时间信息、封面图。
6. 时间信息示例：`04月18日`。
7. 封面图默认取该地点下排序后的第一张照片。

### 交互说明

1. 点击地点卡片，进入该年份 + 地点下的 Photos 页面。
2. 页面顶部展示当前年份。
3. 提供返回按钮，可回到 Years 页面。
4. 底部导航中 Places 高亮。

## 4.3 前台 Photos 页面

### 操作者

普通访问用户。

### 前置条件

1. 用户已选择某个年份。
2. 用户已选择某个地点。
3. 该年份 + 地点下存在照片。

### 需求说明

Photos 页面用于展示具体照片列表，是最终浏览照片的页面。

### 展示规则

1. 页面顶部展示当前年份和地点。
2. 照片不显示文件名。
3. 照片应紧密排列，减少卡片之间的无效空隙。
4. 照片排列顺序优先使用智能排序结果。
5. 若该相册未执行智能排序，则默认按时间排序。
6. 页面应尽量保留当前液态玻璃视觉风格。

### 排列规则

优先级如下：

1. 如果存在 CLIP + TSP 智能排序结果，按智能排序结果展示。
2. 如果不存在智能排序结果，按照片时间从早到晚或从晚到早展示，首版可默认从早到晚。
3. 同一天多张照片可按上传时间或文件创建时间排序。

### 交互说明

1. 点击照片后进入全屏查看。
2. 全屏查看在当前标签页中完成，不打开新标签页。
3. 全屏状态下支持：
   - 查看当前照片；
   - 上一张；
   - 下一张；
   - 点击背景或关闭按钮退出全屏；
   - 再次点击图片可返回 Photos 页面。
4. 返回后保持原 Photos 页面状态，不重新跳回 Years 页面。

## 4.4 照片全屏查看

### 操作者

普通访问用户。

### 前置条件

用户在 Photos 页面点击某张照片。

### 需求说明

提供沉浸式照片查看体验。

### 交互说明

1. 点击照片后，页面出现全屏遮罩层。
2. 当前照片居中显示，保持原图比例。
3. 全屏层中提供：
   - 上一张按钮；
   - 下一张按钮；
   - 关闭按钮；
   - 可选照片序号，例如 `3 / 20`。
4. 切换上一张 / 下一张时不刷新页面。
5. 到达第一张时，上一张按钮可置灰或循环到最后一张。
6. 到达最后一张时，下一张按钮可置灰或循环到第一张。
7. 首版建议使用“不循环，按钮置灰”方案，逻辑更清晰。

## 4.5 后台访问与登录

### 操作者

单管理员。

### 前置条件

管理员知道后台固定隐藏路径和账号密码。

### 需求说明

后台通过固定隐藏路径 `/admin` 访问，前台不展示 Admin 入口。

### 交互说明

1. 用户访问 `/admin`。
2. 如果未登录，展示登录页。
3. 输入账号密码。
4. 登录成功后进入 Admin 后台首页。
5. 登录失败时提示账号或密码错误。
6. 未登录状态下直接访问后台管理接口，应拒绝访问。

### 权限说明

1. 后台仅支持单管理员。
2. 普通前台访问用户无上传、修改、删除权限。
3. 删除照片、修改照片信息、智能排序等操作都只能在登录后台后进行。

### 操作者

单管理员。

### 前置条件

管理员已登录后台。

### 需求说明

后台首页不直接铺开展示所有复杂表单，而是通过按钮触发具体管理操作。

4. 最近上传照片。

## 4.7 添加照片

### 操作者

单管理员。

### 前置条件

管理员已登录后台。

### 需求说明

后台不直接展示添加照片表单，而是展示“添加照片”按钮。点击后，再显示添加照片所需信息。

### 交互流程

1. 管理员进入后台首页。
2. 点击“添加照片”按钮。
3. 页面展开或弹出添加照片面板。
4. 管理员选择照片文件。
5. 管理员填写照片时间，精度为年月日。
6. 管理员填写地点，地点为单一文本。
7. 点击上传。
8. 系统保存照片文件到本地目录。
9. 系统保存照片元数据。
10. 上传成功后，前台 Years、Locations、Photos 页面自动根据数据更新。

### 表单字段

| 字段 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| 照片文件 | 文件上传 | 是 | 支持常见图片格式 |
| 时间 | 日期选择 | 是 | 精度为年月日 |
| 地点 | 文本输入 | 是 | 单一文本，例如“香港” |
| 年份 | 自动生成 | 是 | 根据时间字段自动提取 |
| 文件名 | 系统读取 | 是 | 后台可见，前台不展示 |

## 4.8 修改照片信息

### 操作者

单管理员。

### 前置条件

管理员已登录后台，且图库中已有照片。

### 需求说明

后台不直接展示修改照片信息的表单，而是提供“修改照片信息”按钮。点击后，管理员先选择照片，再进入修改表单。

### 交互流程

1. 管理员点击“修改照片信息”按钮。
2. 系统展示选择区域：
   - 选择年份；
   - 选择地点；
   - 选择照片。
3. 照片选择列表中展示：
   - 缩略图；
   - 照片名称；
   - 当前时间；
   - 当前地点。
4. 管理员点击某张照片。
5. 系统展示修改表单。
6. 管理员修改时间或地点。
7. 点击保存。
8. 系统更新照片元数据。
9. 前台展示结构自动更新。

### 修改规则

1. 修改时间后，照片可能移动到新的年份。
2. 修改地点后，照片可能移动到新的地点分组。
3. 如果原年份或原地点下已无照片，则前台自动隐藏该年份或地点。
4. 修改不会改变照片文件本身。
5. 修改后，相关相册的智能排序状态建议标记为“需重新排序”。

## 4.9 删除照片

### 操作者

单管理员。

### 前置条件

管理员已登录后台，且图库中已有照片。

### 需求说明

仅后台支持删除照片，前台不提供删除能力。

### 交互流程

1. 管理员进入后台。
2. 进入当前图库管理区或修改照片信息区域。
3. 选择某个年份、地点、照片。
4. 点击删除。
5. 系统弹出二次确认。
6. 管理员确认后，系统删除照片文件和照片元数据。
7. 删除成功后，前台自动更新展示结构。

### 删除规则

1. 删除照片需要二次确认。
2. 删除后不可在前台访问该照片。
3. 如果某地点下照片全部被删除，则该地点在前台隐藏。
4. 如果某年份下照片全部被删除，则该年份在前台隐藏。
5. 首版可不做回收站，后续可扩展恢复能力。

## 4.10 CLIP + TSP 智能排序

### 操作者

单管理员。

### 前置条件

1. 管理员已登录后台。
2. 某个年份 + 地点下有至少 2 张照片。
3. 本地环境已具备运行 CLIP 模型和 TSP 排序算法的能力。

### 需求说明

在 Photos 层级中，通过 CLIP 视觉模型提取图片特征，再通过 TSP 算法计算照片排列顺序，使相邻照片之间视觉过渡更自然。实现可参考`travel-shots` 项目思路。

### 交互说明

1. 后台提供“重新排序”按钮。
2. 管理员可选择某个年份 + 地点相册。
3. 点击“重新排序”后，系统异步执行排序任务。
4. 排序执行期间，后台显示“排序中”状态。
5. 排序完成后，后台显示“已排序”状态。
6. 前台 Photos 页面按新的排序结果展示照片。

### 算法逻辑

1. 读取目标相册中的照片。
2. 使用 CLIP 模型提取每张照片的视觉 embedding。
3. 计算照片之间的视觉距离。
4. 使用 TSP 算法寻找视觉距离较优的排列路径。
5. 保存排序结果。
6. 前台读取排序结果并展示。

### 异常处理

1. 如果排序失败，后台展示失败提示。
2. 如果照片数量过少，例如 1 张，则不需要排序。
3. 如果排序中用户继续上传新照片，该相册应标记为“需重新排序”。
4. 如果 CLIP 模型不可用，前台仍可按时间顺序展示照片。

## 5. 非功能需求详述

### 5.1 图片存储

当前阶段图片暂存在本地目录。

建议目录结构：

```text
/uploads
  /photos
    /2015
      /山东
        photo_001.jpg
        photo_002.jpg
```

同时需要保存照片元数据，例如：

```json
{
  "id": "photo_001",
  "filename": "1_香港_20260418.JPG",
  "path": "/uploads/photos/2015/山东/photo_001.jpg",
  "date": "2026-04-18",
  "year": "2026",
  "location": "山东",
  "sortIndex": 1,
  "createdAt": "2026-05-19T09:00:00"
}
```

后续如果迁移到云端对象存储，应尽量不改变前台读取逻辑，只替换图片 URL 生成方式。

### 5.2 性能要求

1. Photos 页面应使用懒加载。
2. 图片应生成缩略图，列表页优先加载缩略图。
3. 全屏查看时再加载较高清图片。
4. CLIP + TSP 排序必须异步执行，避免阻塞后台页面。
5. 如果照片数量较多，应显示排序进度或至少显示“排序中”。

### 5.3 安全要求

1. 前台不展示 Admin 入口。
2. 后台固定路径为 `/admin`。
3. `/admin` 页面必须账号密码登录。
4. 后台接口需要鉴权，不能只隐藏页面。
5. 删除照片需要二次确认。
6. 单管理员账号密码不应硬编码在前端代码中。

### 5.4 兼容性要求

1. 首版优先适配桌面浏览器。
2. 支持 Chrome、Edge、Safari 等主流浏览器。
3. 移动端适配可作为后续优化项。
4. 页面视觉效果应在不支持高级滤镜时有降级方案。
